---------DAY 2----------
Q1. What is Spring Boot and how is it different from Spring Framework?

Spring Framework requires lots of manual XML configuration. Spring Boot uses auto-configuration — it detects what's on your classpath and sets things up automatically. Spring Boot also includes an embedded Tomcat server so you don't need to deploy WAR files separately.


Q2. What does @RestController do? How is it different from @Controller?

@Controller is used for MVC web apps that return HTML views. @RestController = @Controller + @ResponseBody — it automatically serializes (converts) Java objects to JSON and writes them directly to the HTTP response. Use @RestController for REST APIs.


Q3. What is Dependency Injection and why does Spring use it?

Instead of creating objects manually (new ShopService()), Spring creates and manages objects (called beans) and injects them where needed via @Autowired. This makes code loosely coupled — your Controller doesn't need to know HOW ShopService is created, just that it exists. Makes testing and maintenance easier.


Q4. What is pom.xml in Maven?

pom.xml (Project Object Model) is the configuration file for Maven. It defines project identity (groupId, artifactId), Java version, and most importantly — dependencies (external libraries). Maven reads this file and automatically downloads all required JARs from the internet (from Maven Central repository).


Q5. Why do we need @CrossOrigin?

Browsers enforce Same-Origin Policy — they block JavaScript on localhost:3000 from making API calls to localhost:8080 because the ports differ. @CrossOrigin(origins = "http://localhost:3000") adds an HTTP header (Access-Control-Allow-Origin) to the response telling the browser: "Yes, this origin is allowed." Without this, our React app would get a CORS error.


Q6. What is the difference between @GetMapping and @PostMapping?

@GetMapping handles HTTP GET requests — used to read/fetch data. @PostMapping handles HTTP POST requests — used to send/create data. GET requests have no body; POST requests carry data in the request body (like a CSV file or JSON data).


Q7. What are the 3 layers we created and why?

Controller — Handles incoming HTTP requests and outgoing responses (like a receptionist). Service — Contains all business logic (like the actual worker). Model — Defines the data structure/shape (like a blueprint). This is called Separation of Concerns — each layer has one responsibility, making the code easier to maintain and test.


-----------DAY 3-----------
Q1: What is MultipartFile in Spring Boot?
It's an interface that represents a file uploaded via an HTTP multipart request. It gives methods like getInputStream(), getOriginalFilename(), and isEmpty() to work with the uploaded file.
Q2: What is ProcessBuilder and why use it over Runtime.exec()?
ProcessBuilder is the modern way to run external processes in Java. Unlike Runtime.exec(), it lets you configure the environment, redirect error streams, and set working directories more cleanly.
Q3: Why do we save the MultipartFile to a temp file before passing to Python?
MultipartFile is an in-memory/stream object — it doesn't exist as a physical file on disk. Python's open() function needs an actual file path, so we must write it to disk first.
Q4: What does process.waitFor() do?
It blocks the Java thread until the external process (Python) completes. Without it, Java might try to read output before Python has finished writing it.
Q5: Why use redirectErrorStream(true)?
It merges Python's stderr (error output) into stdout (normal output) so we can read both from one stream. Without it, if Python crashes, Java would see empty output and miss the error message.  


-------------DAY 4----------------
Q1: What is the difference between JPA and Hibernate?
JPA is a specification — a set of rules and interfaces written by Java that define how a Java ORM (Object Relational Mapper) should work. Hibernate is a framework that implements those rules. Think of JPA as a job description and Hibernate as the employee doing the actual work. You write code against JPA interfaces, and Hibernate executes the real SQL behind the scenes.

Q2: What does @Entity do?
It marks a Java class as a database entity, meaning Hibernate should map it to a table. Without @Entity, Hibernate ignores the class. With it, every field becomes a column, every instance becomes a row, and operations like .save() translate to SQL INSERT statements.

Q3: What is Dependency Injection and why does Spring use @Autowired?
Dependency Injection means instead of creating your own objects with new, you declare what you need and a framework provides it. @Autowired tells Spring: "I need an instance of this type — please create one and inject it here." This decouples classes from each other and makes code easier to test and maintain.

Q4: What is the difference between save() and findAll() in a JPA Repository?
save(entity) performs a write operation — INSERT if the entity has no id yet, UPDATE if it already has one. findAll() performs a read — it generates SELECT * FROM table and maps each row back to a Java object. Both are provided for free by extending JpaRepository.

Q5: Why do we use @PrePersist?
@PrePersist is a lifecycle callback that fires automatically just before an entity is saved to the database for the first time. It's perfect for setting fields that should always be auto-populated — like createdAt timestamps — so you never accidentally forget to set them before calling .save().

Q6: What is ddl-auto=update and is it safe for production?
ddl-auto=update tells Hibernate to automatically modify the database schema to match your entity classes — creating tables and adding columns as needed. It's convenient for development but not recommended for production because it can make unexpected schema changes. In production, you'd use ddl-auto=none and manage migrations with a dedicated tool like Flyway or Liquibase.

Q7: What does @CrossOrigin do and why is it needed?
@CrossOrigin configures CORS — Cross-Origin Resource Sharing. Browsers block JavaScript on one domain (e.g. localhost:3000) from calling APIs on another domain (e.g. localhost:8080) as a security measure. @CrossOrigin(origins = "*") tells the backend to include headers in its response that tell the browser "this is allowed." Without it, your React frontend would get a CORS error when calling the Spring Boot API.   


---------DAY 5----------------
Q1: What is the difference between state and props in React?
State is data owned and managed by a component itself — it can change, and when it does, React re-renders the component. Props are data passed into a component from its parent — the child reads them but cannot change them. A rule of thumb: state is internal, props come from outside.
Q2: Why do we need the dependency array in useEffect?
The dependency array tells React when to re-run the effect. An empty array [] means "run once after the first render." If you omit it entirely, the effect runs after every render, which can cause infinite loops if the effect itself updates state. If you put a variable in it like [userId], the effect re-runs whenever userId changes.
Q3: Why must you never modify state directly in React?
React's rendering system only triggers a re-render when you call the setter function (like setRules). If you write rules.push(newRule) directly, the array in memory changes but React doesn't know about it and the UI stays stale. The setter function tells React "something changed, please recalculate what to show."
Q4: What is CORS and why does your app need it?
CORS is a browser security feature that blocks JavaScript on one origin (protocol + domain + port) from making HTTP requests to a different origin. React on localhost:5173 and Spring Boot on localhost:8080 are different origins because the ports differ. The backend must send CORS headers (Access-Control-Allow-Origin) to tell the browser the request is permitted. The @CrossOrigin annotation in Spring Boot does this automatically.
Q5: What is FormData and why do you use it for file uploads?
FormData is a browser API that constructs a multipart/form-data request body — the standard format for sending files over HTTP. Regular JSON (application/json) can only carry text. multipart/form-data can carry binary file data alongside text fields. Spring Boot's @RequestParam MultipartFile is designed to receive exactly this format.
Q6: What does async/await do and how is it different from .then()?
Both handle Promises (asynchronous operations that return a value later). .then() chains callbacks. async/await lets you write asynchronous code that looks synchronous — await pauses execution in that function until the Promise resolves. async/await is generally cleaner and easier to read, especially when you have multiple sequential async operations or need try/catch error handling.
Q7: Why does React require a key prop when rendering lists?
When React re-renders a list, it needs to know which items changed, were added, or were removed — so it can update only those DOM nodes instead of re-rendering everything. The key prop is a stable identifier React uses to match old list items to new ones. Without it, React either throws a warning or makes incorrect DOM updates.


-------------DAY 6----------------
Q1: What is multi-tenancy and how are you implementing it in ShopLens?
Multi-tenancy means multiple independent users ("tenants") share the same application and database, but each sees only their own data.
In ShopLens, you implement it at the service layer:

Every table has an owner_id column
Every owner query filters by WHERE owner_id = ?
No owner can ever request another owner's ownerId without being blocked

This is called row-level security — the data isolation is enforced by always filtering queries, not by having separate databases.

Q2: What is RBAC (Role-Based Access Control)?
RBAC means you assign permissions to roles, not to individual users. Then each user gets a role.
In your system:

Role OWNER can: upload CSV, view own data, manage own inventory
Role ADMIN can: view all data, deactivate accounts, delete owners

Benefits over individual permissions:

If you add a new endpoint, you only decide "is this OWNER or ADMIN?" — not "which of the 500 owners can access this?"
Easy to add new roles later (e.g., VIEWER who can see but not upload)


Q3: What is the difference between @ManyToOne and @OneToMany?
@ManyToOne → on the "many" side (the child)
@OneToMany → on the "one" side (the parent)
In your code, AnalysisResult has @ManyToOne Owner owner because:

One owner → many analysis results
Each analysis result → one owner

You could add @OneToMany on the Owner entity too (a list of all their results), but you didn't — and that's fine. You don't always need both sides mapped.

Q4: Why use FetchType.LAZY instead of EAGER?
java@ManyToOne(fetch = FetchType.LAZY)  // only load Owner when you call .getOwner()
@ManyToOne(fetch = FetchType.EAGER) // always load Owner automatically with every query
EAGER is dangerous because:

You query analysisResultRepository.findByOwnerId(5) — gets 200 rows
Each row EAGERLY loads its Owner
That's 200 extra SQL queries running silently — called the N+1 problem
Your app slows down massively

LAZY means: only load the Owner object when the code explicitly calls result.getOwner(). If you never call it, no extra query runs.

Q5: Why do you delete child records before deleting the parent in deleteOwner()?
Because MySQL enforces foreign key constraints. The analysis_results table has:
sqlFOREIGN KEY (owner_id) REFERENCES owners(id)
If you try to delete an Owner row while analysis_results rows still reference it, MySQL throws:
Cannot delete or update a parent row: a foreign key constraint fails
You must delete all child rows first (results, inventory, discounts, etc.) then delete the parent (owner). This is manual cascade delete — in Day 7 you'll learn how to do this automatically with @OneToMany(cascade = CascadeType.ALL).

Q6: Why is EnumType.STRING better than EnumType.ORDINAL?
ORDINAL stores the position: ADMIN=0, OWNER=1.
If a new developer adds a role in the middle:
javapublic enum Role { ADMIN, SUPERUSER, OWNER }
// Now OWNER = 2, but the database still has 1 stored for old owners
// All existing OWNER accounts are now SUPERUSER — catastrophic data corruption
STRING stores "ADMIN" and "OWNER" as text. Adding SUPERUSER in the middle doesn't affect any existing data. Always use STRING.

Q7: What is the N+1 query problem?
If you load 100 AnalysisResult rows with EAGER fetch, JPA runs:

1 query to get all 100 results
100 separate queries to load the Owner for each result

That's 1 + 100 = 101 queries when you only needed 1 or 2. This is the N+1 problem.
Solutions:

Use FetchType.LAZY (your current approach — good for Day 6)
Use @Query with JOIN FETCH to load both in one query
